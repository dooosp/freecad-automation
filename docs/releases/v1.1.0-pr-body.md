# Release v1.1.0 to master: FreeCAD-backed automation contracts, review workflows, and verification lanes

This PR promotes the v1.1 release-candidate line into `master`. It consolidates the shift to a FreeCAD-backed automation pipeline with a separate manufacturing-review layer, brings the new config/output/API contracts and production-readiness workflows onto the default branch, and tightens the release surface so the README, package metadata, CLI/runtime docs, and CI claims all match the verification the repository actually owns.

## Why this PR

The v1.1 line is more than a docs refresh. It changes the repository's public surface, adds new workflow areas, formalizes user-facing contracts, and narrows support claims to source-backed runtime paths. This merge packages those integrated changes into a reviewer-friendly release candidate for `master` without overstating platform support or runtime coverage.

## What changed

- aligned the public surface across `README.md`, `package.json`, CLI/runtime messaging, support docs, and release docs around the project's actual positioning: a FreeCAD-backed automation pipeline plus a plain-Python/Node manufacturing-review layer
- expanded the manufacturing-review and production-readiness workflows, including `review`, `process-plan`, `line-plan`, `quality-risk`, `investment-review`, `readiness-report`, `stabilization-review`, `generate-standard-docs`, `ingest`, `quality-link`, `review-pack`, and `compare-rev`
- added richer checked-in examples, runtime JSON inputs, rule packs/profiles, portfolio-style walkthroughs, cross-site comparison artifacts, and a parameter-sweep gallery so the release has concrete reviewer-visible outputs
- formalized the user-facing config contract around `config_version = 1`, with canonical schema coverage, compatibility aliases, migration warnings, `fcad validate-config`, `fcad migrate-config`, and parity coverage for checked-in examples
- introduced a first-class artifact-manifest contract shared across CLI runs, local API jobs, sweeps, and stdout-oriented commands via `--manifest-out`, with the schema documented and validated in-repo
- added a local/dev-first HTTP API backed by the same service layer and runtime path, including `/health`, `/jobs`, `/jobs/:id`, and `/jobs/:id/artifacts`, plus response schemas and shared runtime diagnostics
- split test lanes into hosted fast lanes and repository-owned runtime smoke, and added release/support/testing docs that describe those boundaries explicitly

## Validation performed

- hosted fast lanes in this branch are explicitly split as `npm run test:node:contract`, `npm run test:node:integration`, `npm run test:snapshots`, and `npm run test:py`
- repository-owned runtime smoke now exists in `.github/workflows/freecad-runtime-smoke.yml` for self-hosted macOS and covers real `fcad check-runtime`, `fcad create`, `fcad draw --bom`, `fcad inspect`, `fcad fem`, and `fcad report`
- the runtime smoke harness validates artifact manifests for `create`, `draw`, `fem`, and `report`, and writes `output/runtime-smoke/smoke-manifest.json` for uploaded evidence
- the same workflow also runs runtime-backed Python smoke regressions for `tests/test_cli_runtime.py`, `tests/test_infotainment_draw_qa_regression.py`, and `tests/test_infotainment_hole_dia_regression.py`
- release-prep docs in this branch now call out the exact final audit set: `npm ci`, `node bin/fcad.js --help`, `npm run test:node:contract`, `npm run test:node:integration`, `npm run test:snapshots`, `npm run test:py`, `node bin/fcad.js check-runtime`, `node bin/fcad.js check-runtime --json`, strict validation of the checked-in v1 examples, and `npm run test:runtime-smoke`, with any local environment gaps recorded explicitly instead of silently treated as passing

## Support / runtime verification boundaries

- hosted CI does not install or launch FreeCAD; it only proves the hosted-safe Node, snapshot, and Python lanes
- repository-owned live runtime verification is self-hosted macOS with FreeCAD 1.1.x
- Linux local with FreeCAD, Windows native + FreeCAD, and WSL -> Windows FreeCAD remain compatibility-only paths that still rely on user-local validation
- `fcad check-runtime --json` and `GET /health` now share the same runtime-diagnostics story so tooling can consume the same support-boundary and remediation data that users see in the CLI

## Config / migration notes

- user-facing configs are now treated as versioned `config_version = 1` documents
- legacy inputs still load, but missing `config_version`, top-level `material`/`process`, `[[operations]].type`, and legacy `export` booleans are treated as compatibility aliases and surface warnings or migration output
- `fcad validate-config` validates user-facing shape plus migration state, and `fcad migrate-config` rewrites toward canonical v1 while preserving compatibility-only fields when removing them would be risky
- checked-in examples are expanded and increasingly used as canonical v1 references, with parity coverage added to keep docs/examples aligned with the schema contract

## Output / API contract notes

- artifact-producing CLI runs, sweeps, and local API jobs now share `schemas/artifact-manifest.schema.json` as the provenance contract
- stdout-oriented commands keep human-readable stdout by default, but `inspect`, `fem`, `tolerance`, and `dfm` can emit a stable manifest with `--manifest-out`
- local API job records persist the same manifest shape and expose it through `GET /jobs/:id` and `GET /jobs/:id/artifacts`
- the local API is intentionally local/dev-first, binds to `127.0.0.1`, and currently covers the real service-layer path for `create`, `draw`, `report`, and `inspect`

## Known limitations

- `tolerance` is still outside the repository-owned runtime smoke lane; it remains a deeper local validation path
- hosted CI should not be read as evidence of live FreeCAD execution on macOS or Linux
- Linux, Windows native, and WSL -> Windows FreeCAD are not elevated to equal-maturity runtime claims in this release
- the manufacturing-review outputs are decision-support artifacts and still require engineering judgment; they are not a substitute for engineering sign-off
