# FreeCAD Automation v1.1.0 Release Notes Draft

Recommended tag: `v1.1.0`

This draft is structured so it can be published with minimal editing.

## Release Summary

FreeCAD Automation 1.1.0 positions the project as a FreeCAD-backed automation pipeline with a separate plain-Python / Node manufacturing-review layer. The release tightens runtime diagnostics, makes support boundaries explicit, and formalizes the user-facing config contract around `config_version = 1`.

The key operational change is that `fcad check-runtime` is now the first installation and troubleshooting command. It reports searched paths, runtime selection, active overrides, detected FreeCAD/Python details, command classes, verification scope, and remediation steps when setup fails.

## What Changed Since 1.0.x

### Product messaging and support boundary

- Public wording now consistently describes the project as a FreeCAD-backed automation pipeline rather than a generic CAD app.
- The repository’s two public layers are called out explicitly:
  - FreeCAD-backed CLI for CAD generation, TechDraw drawings, inspection, FEM, tolerance, and reporting
  - plain-Python / Node layer for DFM, readiness, stabilization review, review-pack generation, and related manufacturing-review workflows
- Support claims are now centralized in [docs/support-matrix.md](../support-matrix.md).

### Runtime diagnostics

- `fcad check-runtime` is now the central installation and diagnostic tool.
- Runtime diagnostics now report:
  - searched candidate paths
  - selected runtime and resolution source
  - active `FREECAD_*` overrides
  - detected FreeCAD/Python details when the runtime can provide them
  - which commands require FreeCAD
  - which commands can run without FreeCAD
  - mixed / conditional commands
  - actionable remediation steps when detection fails

### Config lifecycle

- User-facing configs are now treated as `config_version = 1`.
- `fcad validate-config <path>` validates user-facing config shape plus migration state.
- `fcad migrate-config <path> [--out <file>]` writes a versioned config and summarizes automatic changes, compatibility-only fields, and manual follow-up.
- Legacy unversioned configs still load, but they now emit deprecation warnings so older setups can be upgraded deliberately.

### Verification and release docs

- The support matrix now separates repository-owned verification from compatibility-only paths.
- Testing docs now make it clearer which lanes do and do not launch FreeCAD.
- This release note draft is organized for publication instead of internal patch archaeology.

### Studio tracked execution hardening

- The local/dev-first `fcad serve` path now documents a stricter tracked-job contract for `create`, `draw`, `inspect`, and `report`.
- Browser-visible job payloads are sanitized so artifact-driven reruns expose source metadata without echoing raw executor paths back into the shell.
- Queue controls are explicit and capability-driven: queued cancel is supported, retry is limited to failed/cancelled jobs, and running-job cancellation remains opt-in executor behavior rather than an implied forced-stop claim.
- The studio shell now treats `/jobs` as a multi-job monitor and jobs-center source rather than a single active-run indicator.

## What Is Verified

### Maintainer-local verification

- macOS + `FreeCAD.app` 1.1.x:
  - `fcad check-runtime`
  - `fcad create`
  - `fcad draw --bom`
  - `fcad inspect`
  - `fcad fem`
  - `fcad report`

### Repository-owned CI verification

- Hosted fast lanes:
  - `npm run test:node:contract`
  - `npm run test:node:integration`
  - `npm run test:snapshots`
  - `npm run test:py`
- Self-hosted macOS runtime smoke:
  - `fcad check-runtime`
  - `fcad create`
  - `fcad draw --bom`
  - `fcad inspect`
  - `fcad report`

### What hosted CI does not claim

- Hosted CI does not install or launch FreeCAD.
- Hosted CI does not prove Linux or macOS hosted runtime execution.
- Hosted CI does not establish Windows or WSL runtime ownership.

## Known Limitations

- Repository-owned live runtime verification currently exists only for macOS + FreeCAD 1.1.x.
- Windows native, WSL -> Windows FreeCAD, and Linux are compatibility paths, not equal-maturity release claims.
- `report` remains a FreeCAD-backed path because the macOS fallback still uses the bundled FreeCAD Python.
- `analyze-part` is conditional: it can stay in plain Python mode when context already contains model metadata, but it uses FreeCAD for live model inspection or STEP feature detection.
- `sweep` is conditional: cost-only matrices can stay plain Python, while create/fem/report variants require FreeCAD.
- The manufacturing-review layer produces heuristic engineering artifacts and still requires engineering validation.

## Migration Notes For Older Configs

Existing 1.0.x and older configs should continue to load, but users should expect migration warnings until they adopt the canonical v1 shape.

### Recommended upgrade path

1. Run `fcad validate-config <config.toml>`.
2. Review any deprecated or compatibility-only fields.
3. Run `fcad migrate-config <config.toml> [--out <file>]`.
4. Review the generated summary before checking in the migrated config.

### Common legacy patterns still accepted today

- missing `config_version`
- top-level `material` and `process`
- `[[operations]].type` instead of `op`
- legacy `[export] step = true` / `stl = true` flags instead of `formats = [...]`

### Canonical v1 direction

- add `config_version = 1`
- prefer `manufacturing.material` and `manufacturing.process`
- use `[[operations]].op`
- prefer `export.formats = [...]`

## Installation And Troubleshooting Guidance

Use this block as the release-page quick start:

```bash
npm install
npm link
fcad check-runtime
```

If `check-runtime` does not detect FreeCAD:

- macOS: set `FREECAD_APP="/Applications/FreeCAD.app"` and rerun `fcad check-runtime`
- Linux: set `FREECAD_BIN="/path/to/FreeCADCmd"` and rerun `fcad check-runtime`
- Windows: set `FREECAD_BIN=C:\path\to\FreeCADCmd.exe` and rerun `fcad check-runtime`
- WSL -> Windows FreeCAD: use explicit Windows `FREECAD_DIR`, `FREECAD_BIN`, or `FREECAD_PYTHON`; no default bridge is assumed

If the user only needs the manufacturing-review layer, they can continue with the plain-Python / Node commands while FreeCAD setup is still being fixed.
