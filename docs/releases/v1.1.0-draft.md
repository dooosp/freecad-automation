# FreeCAD Automation v1.1.0 Release Notes

Publish target: GitHub release body for tag `v1.1.0`

## Highlights

- FreeCAD Automation 1.1.0 presents the project as a FreeCAD-backed automation pipeline with two public layers:
  - a FreeCAD-backed CLI for CAD generation, TechDraw drawings, inspection, FEM, tolerance, and reporting
  - a plain-Python/Node manufacturing-review layer for DFM, process planning, readiness review, stabilization review, review-pack generation, and standard-document drafts
- `fcad check-runtime` is now the first installation and troubleshooting command. It reports searched paths, runtime selection, active `FREECAD_*` overrides, detected FreeCAD/Python details, command classes, verification scope, and remediation guidance.
- User-facing configs are now formalized around `config_version = 1`, with `fcad validate-config` and `fcad migrate-config` providing validation and migration support.
- Artifact-producing CLI runs, local API jobs, and sweeps now align on the artifact-manifest output contract documented in [docs/output-contract.md](../output-contract.md).
- Public docs now separate repository-owned verification from compatibility-only paths so release claims stay tied to actual evidence.

## Verified Scope

### Maintainer-local verification

- macOS + `FreeCAD.app` 1.1.x:
  - `fcad check-runtime`
  - `fcad create`
  - `fcad draw --bom`
  - `fcad inspect`
  - `fcad fem`
  - `fcad report`

### Repository-owned CI verification

- Hosted fast lanes:
  - `npm run test:node:contract`
  - `npm run test:node:integration`
  - `npm run test:snapshots`
  - `npm run test:py`
- Self-hosted macOS runtime smoke:
  - `fcad check-runtime`
  - `fcad create`
  - `fcad draw --bom`
  - `fcad inspect`
  - `fcad fem`
  - `fcad report`

### What Hosted CI Does Not Claim

- Hosted CI does not install or launch FreeCAD.
- Hosted CI does not prove Linux or hosted-macOS FreeCAD runtime execution.
- Hosted CI does not establish Windows native or WSL -> Windows runtime ownership.

## Compatibility-Only Paths

- Linux local with FreeCAD installed remains a compatibility path. Runtime discovery exists, but there is no repository-owned live runtime smoke on Linux yet.
- Windows native + FreeCAD 1.1 remains a compatibility path. `FREECAD_*` overrides are supported, but there is no repository-owned live runtime smoke yet.
- WSL -> Windows FreeCAD remains a compatibility path. No default bridge is assumed; users must provide explicit Windows FreeCAD paths.

## Config And Output Contract Notes

- Existing 1.0.x and older configs should continue to load, but legacy fields emit migration warnings until they are updated to the canonical v1 shape.
- Common compatibility-only patterns still accepted today:
  - missing `config_version`
  - top-level `material` and `process`
  - `[[operations]].type` instead of `op`
  - legacy `[export] step = true` / `stl = true` flags instead of `formats = [...]`
- The recommended upgrade path is:
  1. Run `fcad validate-config <config.toml>`.
  2. Review deprecated or compatibility-only fields.
  3. Run `fcad migrate-config <config.toml> [--out <file>]`.
  4. Review the generated summary before checking in the migrated config.

## Installation And Troubleshooting

```bash
npm install
npm link
fcad check-runtime
```

If `fcad check-runtime` does not detect FreeCAD:

- macOS: set `FREECAD_APP="/Applications/FreeCAD.app"` and rerun `fcad check-runtime`
- Linux: set `FREECAD_BIN="/path/to/FreeCADCmd"` and rerun `fcad check-runtime`
- Windows: set `FREECAD_BIN=C:\path\to\FreeCADCmd.exe` and rerun `fcad check-runtime`
- WSL -> Windows FreeCAD: use explicit Windows `FREECAD_DIR`, `FREECAD_BIN`, or `FREECAD_PYTHON`; no default bridge is assumed

If you only need the manufacturing-review layer, you can continue with the plain-Python/Node commands while FreeCAD setup is still in progress.
