# FreeCAD Automation v1.1.0 Release Checklist

## Merge Target

- While `integration/v1.1-hardening` is still open, merge release-prep work into `integration/v1.1-hardening` first.
- Before tagging, merge `integration/v1.1-hardening` into `master`.
- Create the `v1.1.0` tag from the release commit on `master`, not from a side branch.

## Version Confirmation

- Confirm [`package.json`](../../package.json) still reports `"version": "1.1.0"`.
- Confirm the release notes draft title and publish target still reference `v1.1.0`.
- Confirm the README release-surface links still point at the current support matrix, testing docs, config schema docs, output contract docs, and release notes draft.

## What CI Proves

- Hosted CI proves:
  - `npm run test:node:contract`
  - `npm run test:node:integration`
  - `npm run test:snapshots`
  - `npm run test:py`
- Hosted CI does not prove a live FreeCAD install or launch on any platform.
- Self-hosted macOS runtime smoke proves:
  - `fcad check-runtime`
  - `fcad create`
  - `fcad draw --bom`
  - `fcad inspect`
  - `fcad report`

## What Runtime Smoke Proves

- Runtime smoke proves the repository-owned macOS FreeCAD path works end to end for `check-runtime`, `create`, `draw --bom`, `inspect`, and `report`.
- Maintainer-local verification adds evidence for `fem` on macOS + `FreeCAD.app` 1.1.x.
- Runtime smoke does not prove Linux, Windows native, or WSL -> Windows runtime execution.

## What Is Still Compatibility-Only

- Linux local with FreeCAD installed
- Windows native + FreeCAD 1.1
- WSL -> Windows FreeCAD
- Any hosted CI environment that does not install or launch FreeCAD

## Repo Metadata

- Confirm the GitHub repository description matches the v1.1.0 product identity.
- Confirm topics cover the current public surface (`freecad`, CAD, TechDraw, FEM, manufacturing, automation).
- Leave the GitHub homepage blank unless a separate public docs or product URL exists.

## Tag And Release Steps

1. Merge the release-prep PR into `integration/v1.1-hardening`.
2. Merge `integration/v1.1-hardening` into `master`.
3. Pull `master` locally and confirm the release docs are present.
4. Run the smallest final verification set appropriate for the release branch:
   - `npm run test:node:contract`
   - `node bin/fcad.js --help`
   - `npm run test:snapshots`
   - `npm run test:py`
   - `fcad check-runtime`
   - `npm run test:runtime-smoke`
5. Create and push tag `v1.1.0` from `master`.
6. Create the GitHub release using [`docs/releases/v1.1.0-draft.md`](./v1.1.0-draft.md) as the release body.
7. Attach or link any runtime-smoke artifacts if release reviewers need the evidence trail.
